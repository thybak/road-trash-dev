# Candidate roadmap

The roadmap orders risk reduction. It is not a deadline or a license to implement every listed item at once.

| Phase | Primary question | Exit evidence |
| --- | --- | --- |
| [0](phase-0-product-definition.md) | What exactly is one complete race? | Accepted one-page contract |
| [1](phase-1-foundation-input.md) | Can two people reliably control it in both browsers? | Input lab demonstrated in Firefox and Chromium |
| [2](phase-2-road-prototype.md) | Does the pseudo-3D road feel convincing? | Playable 60-second single-rider course |
| [3](phase-3-coop-vertical-slice.md) | Is shared-camera co-op enjoyable? | Complete two-player race loop |
| [4](phase-4-combat.md) | Is one melee attack readable at speed? | Players can predict hit and miss outcomes |
| [5](phase-5-ai-content.md) | Can the system sustain varied sessions? | Repeatable 3–5 minute session |
| [6](phase-6-presentation.md) | Does it feel like a coherent game? | Polished vertical slice |
| [7](phase-7-release.md) | Is it stable and performant across targets? | Reproducible release with no critical browser failures |

## Phase transition rule

Before advancing:

1. Demonstrate the exit criterion.
2. Record automated and manual evidence.
3. Capture unresolved risks.
4. Update decisions affected by the evidence.
5. Choose the smallest next-phase slice.

Work may revisit an earlier phase when evidence invalidates a design assumption.
